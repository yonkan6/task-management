Пітч:
Веб-застосунок, призначений для керування (створення, редагування, виконання) завданнями. Інструмент допомагає учням, студентам, та командам у зручному форматі контролювати наявні дедлайни та підвищувати показники особистої ефективності шляхом впровадження автоматизованих засобів планування. Зі сторони керівної частини (викладачі, керівна команда) це зручний спосіб донести до виконавців актуальні завдання, контролювати їхній статус та своєчасність виконання. Тобто система довзволяє спостерігати зручний та прозорий процес керування завданнями для всіх учасників - від виконавців до керівників -   забезпечуючи організованість, контроль дедлайнів і підвищення ефективності роботи.

Проблема:
Наявність великого обсягу роботи, що важко контролювати, внаслідок чого утворюються затримки в роботі та прострочені дедлайни. Відсутність інструмента для зручного планування та компонування роботи, що передбачає навчальний або робочий процес.

Карта стейкхолдерів:
1) Стекхоледри, що сприйматимуть цінність продукту - власле користувачі веб-застосунку (учні, студенти, викладачі та керівники команд), які використовуватимуть його для планування та відстеження виконання завдань. Для цієї групи важливою є зручність інтерфейсу, швидкість роботи, можливість створювати та редагувати завдання, контролювати дедлайни та статуси виконання.
Очікування: наявність зручного інтерфейсу; швидка робота системи без збоїв; можливість створювати, переглядати та редагувати завдання; переглядати дедлайни й статуси виконання завдань.
Обмеження: обмежений час на роботу із системою;потреба у простоті використання; необхідність у мінімальній кількості кліків для виконання основних дій.
2) Стейкхолдери, що оцінюватимуть досягнення цілей проєкту - замовник, викладач або керівник навчального чи робочого процесу, менеджер проєкту. Вони визначають критерії успішності, контролюють відповідність функціоналу початковим вимогам і приймають рішення, чи готовий застосунок. Їхнє оцінювання охоплює якість реалізації функцій, стабільність роботи та здійснений прогрес відповідно встановлених цілей.

Обмеження:
3) Стейкхолдери, що впливають на реалізацію проєкту через часові, технічні та ресурсні обмеження - команда розробки, адміністратор бази даних, технічний консультант або викладач, постачальники інструментів і сервісів. Ця група визначає та контролює можливості виконання технічних рішень, впливає на строки розробки, вибір технологій і обсяг доступних ресурсів. Їхня участь визначає реалістичність упровадження функціоналу, стабільність системи та надає можливість майбутнього масштабування.

Ціль першої ітерації:
Можливість створювати нове завдання та переглядати його у списку.

Ядро цінності:
Надання користувачу можливість легко та зручно переглянути свої завдання і контролювати їх виконання; змінювати статус виконання.

Ознаки успіху:
1) Користувач може створити завдання менше ніж за 40 секунд, здійснивши декілька кліків.
2) Статус виконання завдання оновлюється у реальному часі без збоїв стабільно протягом тижня користування.
3) Користувач без додаткових інструкцій здатний легко орієнтуватись у інтерфейсі веб-застосунку.

Користувацькі сценарії:
1. Створення нового завдання
Початкові умови: Після входу до системи користувач знаходиться на головній сторінці.
Кроки:
1) Користувач відкриває сторінку зі списками завдань.
2) Обирає потрібний список завдань.
3) Користувач натискає кнопку "Додати нове завдання"
4) Вказує заголовок, опис завдання, прикріплює файли відповідно завдання.
5) Вказує строк здачі (дедлайн).
6) Настискає кнопку "Опублікувати".
Результат: У списку завдань з'явилося нове завдання.
2. Редагування завдання
Початкові умови: Після входу до системи користувач знаходиться на головній сторінці.
Кроки:
1) Користувач відкриває сторінку зі списками завдань.
2) Обирає потрібний список завдань.
3) У списці завдань знаходить потрібне та обирає значок "Додатково" (три крапки, розташовані вертикально).
4) У відкриті панелі обирає опцію "Редагувати".
5) Переходить до панелі редагування.
6) Знаходить розділ "Опис завдання", змінює опис завдання.
7) Знаходить розділ "Дедлайн", змінює встановлений строк здачі (дедлайн).
8) Натискає кнопку "Редагувати".
Результат: У завданні були внесені певні зміни - зміст опису завдання та дата дедлайну.
3. Видалення завдання
Початкові умови: У списку завдань присутнє вже непотрібне завдання.
Кроки:
1) Користувач відкриває сторінку зі списками завдань.
2) Обирає потрібний список завдань.
3) Знаходить непотрібне завдання у списку.
4) Обирає значок "Додатково" (три крапки, розташовані вертикально).
5) У відкриті панелі обирає опцію "Видалити".
6) Натискає "Видалити" у вікні підтвердження.
Результат: Завдання повністю видалилося зі списку.
4. Детальний перегляд завдання
Початкові умови: У списку наявне щонайменше одне завдання.
Кроки:
1) Користувач відкриває сторінку зі списками завдань.
2) Обирає потрібний список завдань.
3) Клікає по потрібному завданню.
Результат: Відкрита сторінка з детальною інформацією, що стосується завдання (опис, дедлайн, прикріплені файли).
5. Позначення завдання виконаним.
Початкові умови: У користувача наявне невиконане завдання.
Кроки: 
1) Користувач відкриває сторінку зі списками завдань.
2) Обирає потрібний список завдань.
3) Знаходить у списку активне завдання та клікає по ньому.
4) У відкритій вкладці детльної інформації натискає на кнопку "Здати завдання".
Результат: Для особи, що виклала завдання, та особи, що завдання виконала, завдання позначено виконаним.
6. Перегляд виконаних завдань
Початкові умови: У користувача вже наявне завдання, позначене як виконане.
Кроки:
1) Користувач відкриває сторінку зі списками завдань.
2) Обирає потрібний список завдань.
3) Обирає вкладку "Виконані".
4) Переглядає історію виконаних завдань.
Результат: Користувач бачить свій прогрес виконання завдань.
